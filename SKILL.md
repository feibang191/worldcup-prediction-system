# 统一球赛预测系统

当用户需要预测足球比赛、分析赔率、构建投注方案、回测模型、或生成赛事报告时，使用本 Skill。

## 触发条件

- 预测比赛结果 / 分析赔率 / 足彩推荐 / 竞彩分析
- 构建/改进预测模型 / 回测历史 / A/B对比
- 蒙特卡洛模拟 / 夺冠概率 / 小组出线
- 投注策略 / EV分析 / Kelly优化 / 串关方案
- 抓取赔率数据 / 多源数据采集
- 生成赛事报告 / 可视化仪表盘

## 核心规则

**不要让模型直接拍脑袋猜比分。** 必须使用结构化流程：

```
数据采集 → 特征工程 → 统计概率模型 → 历史回测 → 校准权重 → AI解释
```

预测概率、赛事模拟和评估指标必须由确定性代码计算。AI 只用于解释已算出的结果。

## 数据补充工作流（2026-06-12补充）

当需要评估比赛但发现数据缺失时，按以下优先级补充：

### 优先级1：footystats_team_power_rating（最快，最可靠）
数据库 `footystats_team_power_rating` 表包含成熟的综合实力分，是最快的补充源。

```python
SELECT overall_power_score, attack_score, defense_score, tempo_score, control_score,
       reliability_adjusted_score, model_confidence, rating_tier
FROM footystats_team_power_rating WHERE team_slug=?
```

如果此表有数据，直接使用 `overall_power_score` + `rating_tier` 更新 `worldcup_team_quant_base_v2` 的 `footystats_power_score`。

### 优先级2：footystats_team_quant_index（补充统计指标）
当需要xG/进球率/胜率等明细指标时查此表：
```python
SELECT goals_scored_per_match_avg, goals_conceded_per_match_avg,
       win_percentage_overall_avg, btts_percentage_avg,
       xg_for_avg_overall_avg, xg_against_avg_overall_avg
FROM footystats_team_quant_index WHERE team_slug=?
```

### 优先级3：其他已缓存表
- `sporttery_matches` — 竞彩比赛信息（含match_id映射）
- `sporttery_pool_odds` — 竞彩各玩法赔率
- `odds_snapshots` — 历史赔率快照
- `agent_browser_500com_extracts` — 500.com浏览器抓取缓存

### 优先级4：外部API（需网络可用）
- Sporttery官方API ✅（2026-06-14验证可达: `webapi.sporttery.cn/gateway/uniform/football/getMatchListV1.qry?clientCode=3001`，Mozilla头+Referer直连，返回22+场赔率HHAD/HHAF）

- Titan007（需browser自动化）

### 体彩6种玩法速查（每次分析必查列表）

当分析一场比赛时，立即列出所有可用玩法，不得只查HAD就下结论：

| 代码 | 玩法 | poolStatus | 说明 |
|:----|:----|:----------:|:-----|
| HAD | 胜平负 | Selling/Define | 最基础，但赔率可能偏低 |
| HHAD | 让球胜平负 | Selling/Define | 有让球盘时价值可能更高 |
| HAFU | 半全场 | Selling | 9合1选项，冷门方向赔率高 |
| CRS | 比分 | Selling | 高赔率但精准命中极难 |
| TTG | 总进球数 | Selling | 0-7+球选项，适合有方向的比赛 |
| (无) | 上下单双 | (无API数据) | 了解即可 |

**铁律**：当HAD赔率太低或无HAD时，**不可直接说"不投"**。必须检查HHAD/HAFU/CRS/TTG是否有价值。

### 关键经验
- **Footystats power_score 是现有数据库中最全面的量化指标**，优先于Elo估算
- 当查询 `worldcup_team_quant_base_v2.footystats_power_score` 为 None 时，先在 `footystats_team_power_rating` 中查找
- 查询时用 `team_slug`（英文slug，如 `canada_national_team`），不是中文名
- 500.com 和 Sporttery 同属中国体彩体系，赔率完全一致，不是独立数据源

## 标准投注策略：均值-方差优化

**所有投注决策必须使用均值-方差优化（Markowitz）分配，禁止凭直觉分配。**

### 核心公式

```
最大化：L = Σ b_i·edge_i - (λ/2)·Σ b_i²·σ_i²
约束：Σ b_i = B（预算用完），b_i ≥ 0

解析解：b_i = (edge_i - μ) / (λ·σ_i²)
```

### 标准流程

1. 列出所有可投选项，计算 edge_i = p_i × d_i - 1
2. 筛选 edge > -10% 的选项
3. 用均值-方差优化分配预算（默认 λ=2.0）
4. 取整到10元，确保总额=预算
5. 加入最优2串1组合（edge最高的跨场组合）
6. 输出表格形式的投注方案

### λ选择标准

| 场景 | λ | 说明 |
|:|:|:|:|
| 默认 | 2.0 | 平衡收益与风险 |
| 高置信度 | 1.5 | 适度激进 |
| 信心不足 | 3.0 | 增加分散 |
| 全负edge | 5.0 | 最小化损失 |
| 连亏3天+ | 5.0 | 止损期 |

### 回测验证（2026-06-11）

| 策略 | 期望收益 | 盈利概率 | Sharpe | ROI |
|:|:|:|:|:|
| 原票: 比分4关 | -171元 | 0% | -0.063 | -85.7% |
| 原票: SPF 2串1 | -60元 | 38.5% | -0.320 | -30.2% |
| **均值-方差: 单注+2串1** | **+10元** | **52.0%** | **+0.079** | **+4.9%** |

### 禁止事项

- ❌ 禁止凭直觉分配金额
- ❌ 禁止全押单一选项（单注≤40%预算）
- ❌ 禁止在全负edge时不调整λ
- ❌ 禁止凭"HAD赔率太低/无HAD"就直接跳过比赛。必须先检查所有6种玩法：HAD、HHAD、HAFU、CRS、TTG、上下单双。即使HAD未开或赔率低，HHAD或CRS/TTG可能有价值（如伊拉克vs挪威无HAD但有HHAD(+2) @2.30）
- ❌ 禁止比分串关作为主要策略（仅允许≤10%预算搏冷）

## 保本收益最大化投注技巧

当用户要求在有限预算内最大化"保本概率"时，使用此方法：

### 核心原理

给定 `n` 个互斥结果，赔率分别为 `d₁, d₂, ..., dₙ`，预算 `B`。  
需要分配 `x₁, x₂, ..., xₙ` 使得**任何命中的结果至少回收 B**，同时**剩余预算投最高赔率选项博超额收益**。

### 标准解法（2结果+1高赔）

```
保本条件: x₁·d₁ ≥ B 且 x₂·d₂ ≥ B  
代入: x₁ + x₂ = B_base（最小化B_base）
解得: x₁ = B/d₁, x₂ = B/d₂, B_base = B/d₁ + B/d₂
剩余: B_extra = B - B_base
```

### 实战模板（以美国vs巴拉圭为例，2026-06-13验证）

SP赔率: H=1.87 D=3.02 A=3.80, 预算100元

```
保本主胜线: 100/1.87 = 53.48元 → 投54元
保本平局线: 100/3.02 = 33.11元 → 投33元  
剩余: 13元 → 投1-1比分@5.45
```

| 中奖场景 | 概率 | 回收 | 盈亏 |
|:--------:|:----:|:----:|:----:|
| 主胜 | 47.4% | 101元 | +1元 |
| 1-1平局 | ~10% | 171元 | +71元 |
| 其他平局 | ~19% | 100元 | 保本 |
| 客胜 | 23.3% | 0元 | -100元 |

**保本概率**: 76.7% | **验证结果**: 实际4-1主胜，回收101元 ✅ 保本

### 适用条件

保本总投入 < B（即 1/d₁ + 1/d₂ < 1。当两个结果去水概率之和<100%时适用）

│ 命中后收益       │ 100-400元           │ 500-6000元       │

**选择规则**：
- 用户说"怎么买"或未指定策略 → 默认分散型(6种玩法)
- 用户说"收益最大化""博冷""高赔" → 收益最大化型
- 用户说"200元怎么配" → 两种方案都给出供选择

## 三维度评分模型（比赛选择决策）

每场比赛从三个维度评分（1-5分），总分≥12分才出手：

### 维度1：基本面（实力分差值）
| FootyStats实力差 | 得分 | 说明               |
|:-----------------|:----:|:-------------------|
| ≥25分            | 5分  | 碾压局,方向明确     |
| 15-24分          | 4分  | 有明显实力优势     |
| 8-14分           | 3分  | 实力占优但非碾压   |
| 3-7分            | 2分  | 接近,主客场定胜负  |
| ≤2分             | 1分  | 五五开,跳过        |

### 维度2：赔率价值（EV信号）
| EV值     | 得分 | 说明                       |
|:---------|:----:|:---------------------------|
| >+5%     | 5分  | 显著正EV,强信号            |
| +2~+5%   | 4分  | 正EV,可出手               |
| -2~+2%   | 3分  | 中性,看其他维度            |
| -5~-2%   | 2分  | 负EV,谨慎                |
| <-5%     | 1分  | 显著负EV,跳过             |

计算方式: 去水概率(WhoScored) × Sporttery赔率 - 1

### 维度3：多源一致性
| 条件                              | 得分 | 说明         |
|:----------------------------------|:----:|:-------------|
| SP+WS方向完全一致,价差<0.1         | 5分  | 市场共识     |
| SP+WS方向一致,价差0.1-0.2         | 4分  | 定价合理     |
| SP+WS方向一致但价差>0.2           | 3分  | 有分歧       |
| SP+WS方向不一致                    | 2分  | 方向不明     |
| 只有一个数据源                    | 1分  | 不确定性高   |

### 决策规则
- 总分 ≥ 12分 → ✅ 可出手
- 总分 9-11分 → ⚠️ 观察(临界点可灵活处理)
- 总分 < 9分 → ❌ 跳过,不硬出

## 多信号评分模型（v6.0+）

当需要从赔率数据直接生成推荐方向（不依赖Elo/xG模型）时，使用多信号评分：

| 信号 | 权重 | 说明 |
|:--|:--:|:--|
| 凯利指数 | 强 | >1.0 正EV确认，<0.95 负信号 |
| 亚盘变化方向 | 中 | 升盘→让球方受热，降盘→资金流出 |
| 上盘水位 | 中 | <0.80 大热，>1.05 遇冷 |
| 赔率离散度CV | 弱 | 离散高→公司分歧大→异常定价可能 |
| 竞彩官方对比 | 中 | 竞彩压低某个方向赔率→看好该方向 |
| 个体公司赔率 | 强 | 某公司赔率显著>平均(>10%)→可能定价误差 |

评分方法：
```
for outcome in ['主胜','平局','客胜']:
    总分 = 凯利分 + 亚盘分 + 水位分 + 离散分 + 竞彩分
    推荐 = max(总分)
```

**局限**：纯赔率信号模型，不含球队基本面。凯利高(=1.094)不一定是正EV信号——需结合该方向的市场概率：低概率(<15%)方向即使凯利>1.0风险也极高。

## 关键Pitfalls

### Pitfall: 同场比赛不同玩法不能混串

用户容易误以为可以将同一场比赛的SPF和TTG串在一起。这是**竞彩硬性规则**：
- ❌ 墨西哥SPF主胜 × 墨西哥TTG 2球 = 非法
- ✅ 墨西哥SPF主胜 × 韩国TTG 2球 = 合法

混合过关的木桶原则：关数上限=所含玩法中最低的。含CRS/BQC→最高4关。

### Pitfall: 平局预测=0（Poisson结构性缺陷）

标准Poisson模型中，p_D 永远不是 p_H/p_A/p_D 三者中的最高值。取 `max(p_H, p_D, p_A)` 作为预测时，平局永远不会被选中。

**根因**：Elo差异导致 p_H ≠ p_A，p_D 不可能成为最高概率。这不是参数问题，是"取最大值"预测逻辑的结构性缺陷。

**解决方案**：GBM平局过滤器。训练5058场洲际杯+国际比赛，当 `gbm_draw_prob > 0.35` 且 `p_D > 0.22` 时预测平局。回测效果：平局预测从0场→25场，准确率44%。

### Pitfall: 平局修正因子不能一刀切，必须按Elo差距递减（2026-06-17校正）

**教训来源**：法国(1810) vs 塞内加尔(1580) Elo差230，我用了全局平局修正×1.44，建议押平局@4.20。实际法国3-1胜，平局修正全错。

**根因**：前16场44%平局率中，绝大多数平局发生在实力相近球队之间（Elo差<50），碾压局的平局率远低于此值。全局修正×1.44在Elo差>200时严重高估平局概率。

**解决方案**：平局修正因子随Elo差距递减，非一刀切：

| Elo差 | 主胜修正 | 平局修正 | 客胜修正 | 说明 |
|:-----:|:--------:|:--------:|:--------:|:-----|
| <50 | ×0.77 | ×1.44 | ×0.83 | 实力接近，全局平率适用 |
| 50-100 | ×0.80 | ×1.25 | ×0.85 | 微有差距 |
| 100-150 | ×0.85 | ×1.10 | ×0.87 | 实力分化 |
| 150-200 | ×0.90 | ×1.00 | ×0.90 | 强队明显占优 |
| >200 | ×0.92 | ×0.85 | ×0.92 | 碾压局，信任强队方向 |

**验证**：法国vs塞内加尔(Elo差230)用×0.85修正后平局概率从33%降至18%，方向回归"法国胜" → 与实际3-1一致。

### Pitfall: Elo与xG不相关（不要用Elo推算xG）

2026-06-14验证：Smodel提供18支球队的Elo和xG对照数据，线性回归 R² = 0.0095（几乎零相关）。

**根因**：Elo反映比赛结果，xG反映比赛过程（射门/控球），两者是完全不同的度量维度。

**不要用Elo线性推算xG**，误差极大。替代方案：用赔率反推隐含xG（`references/smodel-v16-analysis.md`）。

Sporttery竞彩和500.com赔率完全一致（同属中国体彩体系），但与bet365/Oddsportal差异显著。例如墨西哥vs南非：竞彩SPF=1.26/4.45/9.00 vs bet365=1.30/4.20/9.00。

**关键**：Sporttery抽水15-20%（overround），bet365仅5-6%。去水后才能用于模型融合。

### Pitfall: 高确信场次才是投注核心

回测显示：置信度≥60%的场次准确率72.7%（5届WC），而全部场次仅55.2%。只在高确信区间下注，跳过低确信场次。

### Pitfall: 环境修正对2022卡塔尔影响小

2022卡塔尔世界杯场馆有空调，环境修正几乎无效。但2026世界杯（墨西哥城2240m海拔、迈阿密33°C高温）环境修正将有显著影响。

### Pitfall: 主场球队进球预期偏低

2026-06-13两场世界杯验证：
- 美国(51) vs 巴拉圭(43)，实力差8分 → 实际半场3-0全场4-1（预期1-1严重偏低）
- 加拿大(42) vs 波黑(72)，实力差30分（客强主弱）→ 实际1-1（波黑半场领先）
- 校准：实力差<10的主场队，预期进球上调至3.0（原1.5，后调2.5）；实力差>20的弱主，主场加成从15%下调至10%

### Pitfall: 微信文章环境验证绕过

微信公众号文章 `r.jina.ai` / `markdown.new` 统一返回403。

**可用方案**：`scripts/weixin_fetch.py` — HTTP请求 + `Accept-Encoding: identity` 绕过gzip → 提取 `id="js_content"` 区域。

**关键**：必须禁用gzip（`Accept-Encoding: identity`），否则解压失败拿乱码；User-Agent必须用移动浏览器头（含 `MicroMessenger`）。

### Pitfall: 赛后比分验证必须多源交叉确认（重要！）

2026-06-13教训：美国vs巴拉圭赛后，我直接用了WhoScored快照数据（显示2-0），后来南山纠正为4-1。**不要依赖单一数据源**。用户批评后立即去懂球帝搜索确认。

## 竞彩投注实战框架

当用户问"怎么买"时，按此流程：

1. **获取最新赔率**：Sporttery竞彩SPF+RQ + bet365（双源对比）
2. **计算EV**：`EV = 模型概率 × 赔率 - 1`，正EV才考虑
3. **Kelly计算**：`f = (bp - q) / b`，用25%分数Kelly控制风险
4. **串关选择**：只串正EV或高确信选项，2串1赔率=各场赔率相乘
5. **风险分级**：
   - 稳健：拆分投注（主胜60元+让球40元）
   - 平衡：2串1中等赔率
   - 激进：2串1高赔率（客胜×让球负）

**禁止**：承诺稳赚、推荐负EV方案、忽略置信度阈值。

### Pitfall: Sporttery businessDate 是当地时间，非北京时间（2026-06-17校正）

**教训来源**：用户要求分析6月17日(北京时间)的4场比赛（法国vs塞内加尔等），我查了Sporttery API的`businessDate=06-17`场次，取了葡萄牙/英格兰/加纳/乌兹别克这4场。实际这4场是6月18日北京时间才踢的。

**根因**：Sporttery API的 `businessDate` 和 `matchDate` 字段使用**比赛当地日期**（非洲/欧洲时间），不是北京时间。06-17 03:00 BJT = 06-16 21:00 欧洲时间 → Sporttery归档到06-16。

**正确做法**：
- 不能只按 `businessDate` 过滤。要结合 `matchDate`（具体开赛时间）换算时区
- 当用户说"6月17日03:00"时，在Sporttery数据中找 `matchDate` 包含"03:00"或"21:00"的场次
- 验证：同一场比赛在ESPN和Sporttery的日期可能差1天

**修正流程**：
1. 用户指定一个BJT日期 → 列出该BJT日期的所有比赛（用赛程表映射）
